# Agent 协作指南

> 本文档定义了 Chopsticks 项目中 Agent 的协作模式、任务管理流程和开发规范。

---

## 1. 核心原则

1. **主要使用中文** - 所有文档、注释、输出均使用中文
2. **代码和文档不同时修改** - 先完成代码或先完成文档，分开提交
3. **文档编写自动调用 project-wiki** - 文档编写优先使用 project-wiki 技能
4. **代码编写自动调用 go-skill** - 代码编写优先使用 go-skills 系列技能
5. **不询问确认** - Agent Mode 下自行判断并继续，遇到不确定时记录并继续

---

## 2. Agent 工作模式

### 2.1 Agent Mode（默认）

默认工作模式，Agent 自行判断执行：

- **不询问确认** - 直接做出合理假设并执行
- **持续性** - 持续工作直到任务完成
- **自行推断** - 遇到不确定时根据上下文推断最合理的方案

### 2.2 Plan Mode

触发命令：`/plan`

- **先计划后执行** - 在做任何修改前先制定计划
- **输出到计划文件** - 将计划写入 `.trae/documents/` 目录
- **用户确认后执行** - 等待用户确认计划后再开始执行

### 2.3 Spec Mode

触发命令：`/spec`

- **先规范后执行** - 先编写详细的规范文档
- **生成三个文件**：
  - `spec.md` - 功能规范
  - `tasks.md` - 任务清单
  - `checklist.md` - 验收检查清单
- **存放到 `.trae/specs/<功能名>/`** 目录
- **用户确认后执行** - 等待用户确认规范后再开始执行

## 3. 任务管理流程

### 3.1 任务文档存放

临时任务文档存放在 `temp/` 目录下：

```
temp/
├── tasks/      # 任务文档存放
└── reports/    # 报告文档存放
```

> **注意**：`temp/` 目录已被添加到 `.gitignore`，不会进入版本控制。

### 3.2 规范文档存放

规范文档存放在 `.trae/specs/` 目录下，每个功能点一个独立目录：

```
.trae/specs/<功能名>/
├── spec.md       # 功能规范
├── tasks.md      # 任务清单
└── checklist.md  # 验收检查清单
```

### 3.3 执行计划文档

临时执行计划存放在 `.trae/documents/` 目录下。

### 3.4 任务执行流程

```
1. 理解任务需求
2. 分析现有代码和文档
3. 调用相关技能（go-skill 或 project-wiki）
4. 制定实施方案
5. 执行任务
6. 更新相关文档
7. 验证代码和测试
```

### 3.5 注意事项

- 每个任务都应创建独立的分支
- 任务文档仅用于临时记录，不提交到 git
- 合并到主干前确保代码通过测试
- 默认不进行推送，除非强调

## 4. 代码编写规范

### 4.1 代码编写流程

1. **使用 go-skills** - 编写代码时自动调用相关 Go 技能
2. **遵循代码设计** - 参考 wiki/developer/DEVELOPER.md 中的设计规范
3. **以文档驱动开发** - 根据架构文档和设计文档编写代码
4. **更新开发进度** - 代码完成后更新相关文档
5. **编译验证** - 编译代码为当前平台的可执行文件 `bin/chopsticks.exe`

### 4.2 Go Skills 技能列表

编写代码时可使用的 Go 技能：

| 技能                  | 用途             |
| --------------------- | ---------------- |
| go-style-core         | 核心 Go 风格原则 |
| go-naming             | 命名规范         |
| go-packages           | 包组织与导入     |
| go-error-handling     | 错误处理模式     |
| go-concurrency        | 并发编程         |
| go-context            | Context 使用     |
| go-control-flow       | 控制流惯用语     |
| go-data-structures    | 数据结构         |
| go-interfaces         | 接口设计         |
| go-defensive          | 防御性编程       |
| go-documentation      | 文档编写         |
| go-testing            | 测试模式         |
| go-performance        | 性能优化         |
| go-linting            | Linting 配置     |
| go-functional-options | 函数式选项模式   |
| go-code-review        | 代码审查         |

### 4.3 添加/修改功能流程

1. 理解要处理的功能
2. 根据文档进行实现
3. 对相关代码做出同步更改
4. 检查新编写的代码问题并修复

---

## 5. 文档编写规范

### 5.1 文档编写流程

1. **使用 project-wiki** - 调用 project-wiki 技能辅助文档编写
2. **使用 go-skill 辅助** - 必要时调用 go-documentation 等技能
3. **注意文档长度** - 复杂文档进行拆分
4. **注意目录结构** - 保持清晰的文档目录结构
5. **保持一致性** - 功能文档、需求文档、架构文档、接口文档、数据文档等保持一致

### 5.2 功能修改流程

1. 先理解修改内容
2. 调用技能生成修改方案
3. 查找当前项目相关文档
4. 对 wiki 中所有相关文档进行覆盖式修改

### 5.3 文档分类

| 类型       | 位置            | 说明           |
| ---------- | --------------- | -------------- |
| 用户文档   | wiki/user/      | 面向最终用户   |
| 开发者文档 | wiki/developer/ | 面向开发者     |
| 设计文档   | wiki/design/    | 架构和设计说明 |
| 参考文档   | wiki/reference/ | 研究和参考资料 |

---

## 6. 提交流程

### 6.1 提交修改内容

1. 根据修改内容进行总结，分成不同步骤
2. 根据内容分步骤暂存修改内容，逐步进行提交
3. 默认不进行推送，除非强调

### 6.2 更新版本

1. 参考 ROADMAP.md，根据版本规划进行更新
2. 更新 CHANGELOG.md
3. 添加 tag
4. 默认不进行推送，除非强调

---

## 7. 编译构建

### 7.1 编译命令

```bash
go build -o bin/chopsticks.exe ./cmd/main.go
```

### 7.2 测试命令

```bash
go test ./...
```

### 7.3 Lint 命令

```bash
golangci-lint run
```
